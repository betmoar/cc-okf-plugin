# OKF v0.2 — "Honest Pitch" Design Spec

Date: 2026-06-19
Status: approved (design); not yet implemented
Branch: `review/v0.1.0`
Driven by: `docs/REVIEW-0.1.0.md`

## Goal

Close the gap between what OKF v0.1.0 promises and what it delivers, without
re-founding the format. Three changes make the existing pitch true:

1. **One link graph, no drift** (review §3) — body `[[id]]` becomes the single
   source of truth; frontmatter `links:` becomes a generated field.
2. **`/okf:rename`** (review §4) — makes the "machine-readable link graph" claim
   real by giving renames an atomic-ish, link-aware operation.
3. **`/okf:query`** (review §2) — makes "durable knowledge" mean *recall*, not
   just "grep the folder."

Plus the positioning paragraph (review §1/§6) so a reader knows when to reach for
OKF over Obsidian or Claude Code's built-in memory.

### Non-goals (explicitly out of scope)

- **Activation-gate reliability** (review §5) — separate investigation; not
  touched here.
- **Graph traversal in query** (neighbors/orphans) — deferred to v0.3. v0.2 query
  is filter-only.
- **Re-founding OKF on an agent-native curation model** — considered and rejected
  for v0.2 (that was the "Re-found" option; we chose "Make the pitch honest").

## Principles carried from v0.1

- Skill = knowledge (no side effects). Commands = actions that run scripts.
  Scripts shared via `${CLAUDE_PLUGIN_ROOT}`.
- Python 3 standard library only. PyYAML used if present, built-in fallback
  parser otherwise.
- Dormant-by-default: nothing here changes the activation model.

## Component A — Link-graph deduplication

### Format change

`links:` is redefined from an author-maintained field to a **generated** field,
derived from the body's `[[id]]` wiki-links. The body is the only place an author
expresses a link. This is documented in `references/spec.md` and the skill.

Rationale for keeping `links:` at all (rather than dropping it): external tools
can read the graph from frontmatter without a markdown-body parser. It is
regenerated, never hand-edited — same contract as the `index.md` table.

### `reindex.py` changes

`reindex` gains a second responsibility beyond regenerating `index.md`: for each
concept file it rewrites the `links:` frontmatter to match the body.

Algorithm per concept:
1. Parse the body, extract `[[id]]` targets via the existing `WIKILINK_RE`.
2. Dedup, preserving first-seen order.
3. Compute the desired `links:` value (a list, or absent when empty).
4. **Surgically splice** only the `links:` block in the frontmatter — replace the
   existing `links:` line(s) in place, or insert/remove the field. Every other
   byte of the file is left untouched.
5. If the spliced result equals the original file, **do not write** (no mtime
   churn, no needless git noise).

Hard constraints:
- Never re-serialize the whole frontmatter (would reflow user formatting).
- Never modify `updated` — `links:` is derived, not an authored edit.
- Empty link set → the `links:` field is **omitted**, not written as `links: []`.

**Splicer contract — the input forms `splice_links` MUST handle** (and the only
region it may touch is the `links:` field itself):
- field absent → insert it (only when the link set is non-empty);
- inline form `links: [a, b]`;
- block form (`links:` followed by `  - a` / `  - b` lines);
- existing empty `links: []` → remove it when the body has no links;
- trailing whitespace / blank lines around the field.
Comments inside a `links:` block (e.g. `# do not reorder`) are **not preserved** —
`links:` is regenerated wholesale, so authors should keep commentary in the body,
not in the generated field. This is a documented consequence of the field being
generated, not a bug.

**First-run migration notice (data-loss guard).** A v0.1 bundle may have
hand-authored `links:` entries with no backing body `[[id]]`. The new contract
drops them — correct, but silently lossy for upgraders. So when the old `links:`
set is **not a subset** of the body-derived set, print a one-line, non-blocking
notice per file: `concepts/foo.md: dropped non-body links [bar, baz] — re-add via
[[bar]]/[[baz]] in the body to keep them`. No prompt, no abort — just visible.

**`--dry-run` flag.** `reindex --dry-run` computes every splice and prints the
per-file change summary (and the migration notices above) **without writing**.
Cheap insurance for the one risk this design names — concept-file mutation — and
the recommended first move on a v0.1→v0.2 upgrade.

**Known limitation — code fences are not parsed.** `WIKILINK_RE` matches
`[[id]]` even inside fenced code blocks. Under v0.2 that means such a token is
treated as a real link (polluting `links:`, and a dangling one is now an ERROR).
v0.2 does **not** strip fences; avoid `[[…]]` in code samples or escape it.
Proper fence-aware extraction is a v0.3 item.

This is the one place in v0.2 where tooling writes to a user's *knowledge* (not
just `index.md`). The splice-and-skip-if-unchanged approach (plus `--dry-run`) is
the mitigation; tests (Component F) pin the exact byte behavior.

### `validate.py` changes

Severity realignment to reflect "body is canonical":
- Dangling `[[id]]` in a body (target id does not resolve) → **ERROR**
  (was WARN). The body is now the source of truth, so a broken link is a real
  conformance failure.
- Frontmatter `links:` out of sync with the body's wiki-links → **WARN**
  ("links: out of date; run /okf:reindex"), mirroring the existing index-staleness
  warning. The generated field being stale is advisory, not fatal.
- The old "frontmatter `links:` target does not exist → ERROR" path is removed:
  `links:` is generated, so a correctly generated file cannot dangle there; if it
  is stale it is caught by the sync WARN above.

## Component B — `/okf:rename`

New command `commands/rename.md` → `scripts/rename.py`.

Invocation: `/okf:rename <old-id> <new-id>` (optional trailing `[bundle-path]`,
default `.`).

**Step order matters for recovery.** The file rename is the **last** mutating
step, so every interrupted state still has `concepts/<old-id>.md` on disk and is
completable by simply re-running the command. (`os.rename` is a single atomic
POSIX syscall, so the rename itself has no partial state — once it succeeds, the
operation is done.) Every step before it is **idempotent**.

Steps:
1. **Pre-flight (no writes until all pass):**
   - `new-id` matches kebab-case `^[a-z0-9][a-z0-9-]*$`.
   - `concepts/<old-id>.md` exists.
   - `concepts/<new-id>.md` does **not** exist (no collision).
   - `old-id != new-id` (else: no-op error).
   No resume-state special case is needed: because rename is last and atomic, the
   only post-failure state is "`<old-id>.md` still present, some idempotent work
   done," which satisfies plain pre-flight and re-runs cleanly.
2. Rewrite `[[old-id]]` → `[[new-id]]` in the bodies of **all** concept files
   (including `<old-id>.md` itself). Idempotent: already-rewritten files are
   no-ops. The substitution is a **literal `[[old-id]]`-token replacement**
   (brackets included), so `[[old-id-v2]]` and similar are untouched — same
   token semantics as `WIKILINK_RE` extraction.
3. Rewrite the `id:` field inside `concepts/<old-id>.md` (still at the old path)
   to `new-id`. Idempotent: setting it to `new-id` when already `new-id` is a
   no-op.
4. Run the reindex routine (regenerate `links:` across the bundle + `index.md`).
5. **Last:** `os.rename(concepts/<old-id>.md, concepts/<new-id>.md)`.

Transactionality: not a true transaction, but fully pre-checked **and**
recoverable by re-run. The worst-case partial state is "bodies/id/links updated
but the file is still named `<old-id>.md`" — re-running `/okf:rename <old> <new>`
completes it. If an OS-level error occurs, the command reports the partial state
and the exact re-run command.

`updated` is **intentionally not modified** by rename (consistent with "`updated`
is human-authored, not tooling-touched"). A user who wants the rename reflected as
an update edits `updated` by hand and runs `/okf:log`.

Logging: `rename.py` does **not** auto-append to `log.md` (consistent with
`reindex`, which also doesn't). The command output tells the user to run
`/okf:log "renamed X to Y"`.

`allowed-tools`: `Bash(python3:*)`, `Read`. No `Edit` — the script performs all
writes itself via Python. (`python` alias dropped for determinism — see §G.)

## Component C — `/okf:query`

New command `commands/query.md` → `scripts/query.py`.

Invocation: `/okf:query [--tag T] [--type Y] [--status S] [--text "..."] [bundle-path]`.

Semantics:
- Filters combine with **AND**. Absent filter = no constraint on that axis.
- `--text` is a case-insensitive substring match over `title` + body.
- `--tag` matches if the tag is present in the concept's `tags` list.
- `--type` / `--status` are exact matches against the frontmatter values.

Output: a list grouped/sorted by (`type`, `id`), each row showing
`id`, `title`, `type`, `status`, `tags`, `file`. Prints a count. Exit 0 always
(an empty result is not an error).

No new dependencies. Reuses the shared frontmatter parser.

`allowed-tools`: `Bash(python3:*)`, `Read`. (`python` alias dropped — see §G.)

## Component D — Shared module `scripts/okf_common.py`

`validate.py` and `reindex.py` today each carry a copied `_coerce`,
`_fallback_parse`, and `parse_frontmatter`. Adding `query.py` and `rename.py`
would make four copies. Extract into one module:

- `parse_frontmatter(text) -> dict | None`
- `body_of(text) -> str`
- `_coerce`, `_fallback_parse` (internal)
- **New:** `splice_links(text, links: list[str]) -> str` — the surgical
  frontmatter splicer used by reindex and rename.
- **New:** `extract_wikilinks(body) -> list[str]` — dedup, ordered.

All four scripts import it. Because scripts are invoked as
`python3 ${CLAUDE_PLUGIN_ROOT}/scripts/<x>.py`, each adds a
`sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))` guard before
`import okf_common` so the sibling module resolves regardless of cwd.

This is a targeted refactor justified by the new code (improving the code we're
working in), not an unrelated cleanup. The four scripts' public CLI behavior is
unchanged except where Components A–C specify.

## Component E — Docs & positioning

- **README:** add a short "When to reach for OKF" paragraph distinguishing it from
  (a) Obsidian/Foam/Dendron + git, and (b) Claude Code's built-in memory store.
  The honest line: OKF is project-scoped knowledge committed to *this repo's* git,
  maintained by Claude as it works — vs. a personal vault (Obsidian) or
  cross-session assistant memory (Claude memory).
- **`skills/okf/SKILL.md`:** `links:` is generated; body `[[id]]` is canonical;
  document `/okf:rename` and `/okf:query` in the lifecycle-commands list.
- **`references/spec.md`:** update the link-graph and frontmatter sections.
- **`references/conformance.md`:** update link rules and severities (dangling
  `[[id]]` = ERROR; `links:` sync = WARN).
- **`references/examples.md`:** reflect generated `links:` in the worked example.
- **`CHANGELOG.md`:** `0.2.0` entry.
- **`.claude-plugin/plugin.json`:** bump `version` to `0.2.0` — **as the final
  ship step only**, after everything else lands and tests pass.

## Component F — Tests

There is no test harness in v0.1. Add one, built test-first for the risky parts.

- `tests/fixtures/bundle/` — a small valid OKF bundle: a few concepts with
  cross-links (including a body wiki-link with no matching `links:` yet, to
  exercise generation), plus `index.md`/`log.md`.
- `tests/` runner — plain-python (stdlib `unittest`) so it needs no extra deps,
  runnable as `python3 -m unittest discover tests`.

**Run the whole suite under both parser paths.** The fallback parser is a real
code path on minimal installs, and the splicer operates on raw text consumed
alongside parsed data — so verifying only the PyYAML path leaves splice behavior
half-tested. Run the suite twice: once normally, once with an `OKF_NO_YAML=1`
env var that forces `okf_common.parse_frontmatter` to skip `import yaml`.

Coverage:
- **Dedup round-trip:** reindex a fixture; assert `links:` matches body
  wiki-links; reindex again; assert the file is byte-identical (idempotent, no
  spurious write).
- **Splice safety:** a concept with custom frontmatter ordering/comments keeps
  every non-`links:` byte after reindex; cover all input forms in the §A splicer
  contract (absent / inline / block / empty `[]` / trailing whitespace).
- **Empty links:** a concept with no body links has no `links:` field after
  reindex.
- **`--dry-run`:** reindex `--dry-run` writes nothing (tree byte-identical) but
  prints the change summary and any migration notices.
- **Migration notice:** a fixture with a hand-authored `links:` entry not backed
  by a body `[[id]]` emits the drop notice on first reindex.
- **Rename happy path:** `[[old]]` rewritten to `[[new]]` everywhere; `id:` and
  filename updated; `index.md` regenerated.
- **Rename recovery (the §B blocker):** simulate a crash *before* the final
  `os.rename` (run steps 2–4, skip step 5); re-invoke `/okf:rename <old> <new>`;
  assert the final state is correct and byte-clean. Also assert pre-flight
  collision and missing-source errors return non-zero **without writing**.
- **Rename token-exactness:** `[[old-id-v2]]` is left untouched when renaming
  `old-id`.
- **Query filters:** tag/type/status/text and their AND-combination return the
  expected ids; empty result exits 0.
- **Validate severities:** dangling `[[id]]` is ERROR; stale `links:` is WARN.

## Build sequence

1. **Component D** (shared module) + tests scaffold — foundation everything imports.
2. **Component A** (reindex generation + validate severities) test-first — the
   riskiest change; lock byte behavior before anything depends on it.
3. **Component B** (`/okf:rename`) — depends on A's splice + reindex.
4. **Component C** (`/okf:query`) — independent of A/B, can parallelize after D.
5. **Component E** (docs) — after behavior is final.
6. **Version bump to 0.2.0** — the single, final ship action, gated on green tests.

## Component G — Conventions & environment

- **`python3`-only in `allowed-tools`.** New commands declare `Bash(python3:*)`
  only, not `Bash(python:*)`. The `python` alias is frequently Python 2 or absent;
  the scripts are `#!/usr/bin/env python3` and invoked as `python3 …`, so the
  alias adds nondeterminism for no gain. The existing v0.1 commands
  (`validate`/`reindex`/`log`/`activate`) are aligned to `python3`-only in the
  same pass — a one-line tidy-up, since Component E already touches the command
  docs.
- **Single-writer assumption.** Reindex and rename assume one writer. Two
  concurrent sessions reindexing/renaming the same bundle can interleave writes
  and corrupt frontmatter. v0.2 does not lock; documented as a known constraint
  (locking is out of scope).

## Risks & the one most likely to bite

- **Reindex writing to concept files** (Component A) is the central risk: a bug in
  the splicer corrupts a user's knowledge, not just a regenerable index. Mitigated
  by surgical splice + skip-if-unchanged + `--dry-run` + idempotency test. **This
  is the claim I'd most expect to need hardening** — the splice must survive every
  valid form in the §A splicer contract (absent / inline `[a, b]` / block list /
  empty `[]` / comments / trailing whitespace).
- **`sys.path` import of `okf_common`** under the real plugin-root invocation —
  verify against `${CLAUDE_PLUGIN_ROOT}/scripts/...`, not just local `cd scripts`.
- **Rename recovery** — resolved by making the file rename the last, atomic step
  (§B): every interrupted state retains `<old-id>.md` and completes on re-run. The
  pinned recovery test (Component F) guards against regression.
- **First-run data loss on upgrade** — hand-authored v0.1 `links:` not backed by a
  body `[[id]]` are dropped; mitigated by the migration notice + `--dry-run`
  (Component A), not by silent preservation.
